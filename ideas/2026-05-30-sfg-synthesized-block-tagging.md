# Enhancement: SFG Phase 1b — Complete Multi-Driver Coverage via Synthesized-Block Tagging

**Date:** 2026-05-30
**Status:** Proposal — ready for team discussion
**Grounded in:**
- Issue #375 (multi-driver detection, C-seq case still open)
- Finding #12 from `ideas/2026-05-29-code-review-findings.md`
- PR #470 (SFG Check 1, shipped — CombBlock + InstConn coverage)
- `ideas/2026-05-28-signal-flow-graph.md` (deferred RegBlock/LatchBlock/Thread surface)

---

## Problem

PR #470 shipped SFG Check 1 for `CombBlock` and `InstConn` nodes, closing most
of issue #375. But `RegBlock`, `LatchBlock`, and `Thread` nodes are intentionally
skipped in `src/signal_flow.rs`:

```rust
ModuleBodyItem::RegBlock(_)   => HashMap::new(),
ModuleBodyItem::LatchBlock(_) => HashMap::new(),
ModuleBodyItem::Thread(_)     => HashMap::new(),
```

The skip comment explains why: the TLM inline-thread lowering pass generates
multiple `RegBlock`s that **legitimately** share register assignments, gated by
synthesized state conditions. A naive driver-count check fires false positives
on every TLM design.

The result is that the **C-seq repro from issue #375** — two user-written `seq`
blocks driving the same register — is still not caught by `arch check`. Users get
a downstream Verilator error ("multiple drivers") instead of a precise source-level
diagnostic. Finding #12 in the 2026-05-29 code-review pass explicitly calls this
out as an untracked gap.

### Concrete example (still uncaught today)

```arch
module M
  port clk: in Clock;
  port rst: in Reset;

  reg counter: UInt<8> reset rst => 0;

  seq on clk rising     // block A
    counter <= counter + 1;
  end seq

  seq on clk rising     // block B — latent bug, currently invisible to arch check
    counter <= 0;
  end seq
end module M
```

`arch check` passes; Verilator later errors with a multi-driver at the SV level,
pointing into generated code rather than the source.

---

## Root Cause

The SFG build pass has no way to distinguish user-written `RegBlock` / `Thread`
nodes from those synthesized by compiler lowering passes. Without that
distinction, any driver-count check on these node types produces false positives
on TLM-heavy designs.

---

## Proposal

### 1. Tag synthesized AST nodes at creation time

Add a `synthesized: bool` field to `RegBlock`, `LatchBlock`, and `Thread` nodes:

```rust
// src/ast.rs
pub struct RegBlock {
    pub on_clause: OnClause,
    pub stmts:     Vec<SeqStmt>,
    pub span:      Span,
    pub synthesized: bool,   // true ↔ generated by a compiler lowering pass
}

pub struct Thread {
    pub name:        Option<Ident>,
    // … existing fields …
    pub synthesized: bool,
}
```

All parser paths set `synthesized: false` (the default). All compiler lowering
passes that emit these nodes set `synthesized: true`. Primary call sites:

| Pass | Location | Nodes it synthesizes |
|------|----------|----------------------|
| Thread FSM lowering | `src/lower_threads.rs` | `RegBlock` (state reg), `Thread` sub-states |
| TLM inline-thread lowering | `src/elaborate.rs` | `RegBlock` (multiple per initiator, sharing state) |
| Lock arbitration lowering | `src/elaborate.rs` | `RegBlock` (arb state), `Thread` (arb thread) |
| Generate-for expansion | `src/elaborate.rs` | `RegBlock` (loop-body), per-element `Thread` |

A one-time audit of every `RegBlock { … }` / `LatchBlock { … }` constructor site in
`src/` identifies which need the `synthesized: true` flag. The compiler will fail
to build if any site omits the new field (Rust struct update syntax catches all
call sites at compile time).

### 2. Include tagged nodes in multi-driver collection with a guard

Update `src/signal_flow.rs::collect_module_drivers` to handle `RegBlock` and
`Thread` items, using the `synthesized` flag as a filter:

```rust
ModuleBodyItem::RegBlock(rb) if !rb.synthesized => {
    collect_seq_block_drivers(rb, module_idx)
}
ModuleBodyItem::Thread(t) if !t.synthesized => {
    collect_thread_drivers(t, module_idx)
}
// synthesized=true nodes still return empty — same as today
ModuleBodyItem::RegBlock(_) => HashMap::new(),
ModuleBodyItem::Thread(_)   => HashMap::new(),
```

The multi-driver check (`check_multi_driver`) requires no changes — it already
compares driver-set sizes and reports conflicts. Two user-written `RegBlock`s
driving the same register will now produce the same `CompileError::MultipleDrivers`
that two `CombBlock`s already produce.

### 3. Wire-to-let promotion for formal (bonus unlock)

`ideas/2026-05-28-signal-flow-graph.md` notes that the formal pass in
`src/formal.rs` could promote `WireDecl` nodes to `let` bindings when all their
drivers are comb-only (enabling hierarchical formal v1 for lock-based designs,
issue #383). The synthesized flag makes this query cheaper: a `WireDecl` emitted
by the lock-arbitration lowering (`synthesized: true`) can be unconditionally
inlined; no driver-set walk needed.

## Tests

Minimum test matrix to pin intent and prevent regressions:

| Test | Expected result |
|------|----------------|
| Two user-written `seq` blocks driving the same `reg` | `CompileError::MultipleDrivers` |
| Two user-written `thread` bodies both writing the same `reg` | `CompileError::MultipleDrivers` |
| Single `seq` block with a `default` + conditional override | No error |
| TLM design with multiple synthesized `RegBlock`s sharing state reg | No error (synthesized flag suppresses) |
| `generate_for` expansion producing multiple `RegBlock`s | No error (synthesized flag suppresses) |

The first two are the exact repros from issue #375 C-seq / C-thread that were
explicitly deferred by PR #470. The last two are the false-positive guards that
justify the original deferral.

---

## Why this matters

- **Closes issue #375 completely.** C-seq and C-thread are the remaining
  multi-driver classes that fall through to confusing downstream Verilator errors.
- **Earlier diagnosis.** A precise `CompileError::MultipleDrivers` with source
  spans is far more actionable than a Verilator net-elaboration error pointing
  at generated SV.
- **Unlocks SFG Check 3.** The sole-writer query needed by the `wait until` exit-fold
  fix (issue #306) requires knowing which blocks drive a register. The synthesized
  flag makes this query conservative: a synthesized block driving `X` is not a
  "user-owned write" that blocks the fold.
- **Safe to land.** The change is additive: all current `arch check` passes remain
  passes; only previously-undetected multi-driver conflicts become new errors.
  No user-visible language semantics change.

---

## What this does not do

- Does not fix `LatchBlock` multi-driver (latch blocks are rarely user-written;
  deferred until a concrete repro surfaces).
- Does not implement the wait-exit fold for issue #306 — it is a prerequisite
  for that fix, not the fix itself.
- Does not change emitted SV.
