## 1. Systemic: bare vs param-aware width/const evaluators in `sim_codegen`

**Pattern.** PR #427, then #439 and #442 in this window, all fix instances of
the same class of bug: a call-site uses `type_bits_te(...)` or
`eval_const_expr(...)` where it should use the `_with_params` variant. The
non-param-aware form silently falls back to default widths (often 32) or
zero-element fan-out when the type/size depends on an enclosing-construct
parameter.

The bare functions delegate to the param-aware ones with an empty params slice
(`src/sim_codegen/mod.rs:4163`), so there is no semantic gap — only an
ergonomic one. Every new fix is a one-by-one substitution.

**Latent sites still on the bare API** (audit, not yet broken in tests):

- `src/sim_codegen/fsm.rs:236-241` — FSM `fsm_widths` map. Direct extension of
  #439's fix; any FSM with `port x: out UInt<W>` where `W` is a param has the
  wrong width recorded.
- `src/sim_codegen/fsm.rs:282` — FSM Vec-reg reset init. Direct twin of #442
  for the FSM lowering path: `reg cnt: Vec<UInt<8>, N>` with `param N` resets
  zero elements.
- `src/sim_codegen/mod.rs:6799, 6821` — coverage toggle counters. Param-sized
  regs get `bits=0` and the toggle-counter branch is silently skipped
  (coverage hole, not correctness).
- `src/sim_codegen/mod.rs:4821` — struct-field widths. `UInt<PARAM>` fields
  lose width.
- `src/sim_codegen/pipeline.rs:38, 48, 72, 135, 155, 201, 275, 375, 410` — all
  use the bare form for pipeline reg/let/port widths and counts.

**Proposal.** Two options, smallest first:

1. **Deprecate the bare API.** Mark `type_bits_te` / `eval_const_expr` as
   `#[deprecated]`, audit each remaining call-site, and route through the
   `_with_params` form with explicit `&m.params` (or
   `&construct.common.params`). Single PR, mechanical, reviewable. Catches all
   listed sites in one pass.
2. **Change the type signature.** Make the functions take
   `Option<&[ParamDecl]>` and require every `None` to be justified with a
   comment. More invasive but harder to regress.

Either option closes the recurring-fix loop documented in
`memory/project_session_2026_04_23_summary.md`.

**Suggested adjacent test that would catch the broadest swath at once.** An
`fsm` with `param N: const = 96` driving `reg buf: Vec<UInt<N>, 4>`, a
`port out_word: out UInt<N>`, and a Vec output port — exercises FSM widths,
FSM Vec reset init, and Vec fan-out in one fixture.

## 2. Arbiter sim cycle-1 off-by-one (RR/LRU)

**Setting.** PR #443 fixed `mutex<priority>` in the native sim (it was
rotating, not picking lowest index) and added `mutex<round_robin>`.

**Finding.** RR/LRU first-cycle-after-reset semantics diverge between
generated SV and native sim.

- SV `emit_arbiter_round_robin` (`src/codegen/arbiter.rs`) starts the scan
  *at* `rr_ptr_r` (initially 0): on cycle 1 with `req[0]=req[1]=1`, idx 0
  wins.
- Native sim (`src/sim_codegen/mod.rs:8388-8463`) starts the scan at
  `_last_grant + 1` (initially 1): on the same cycle, idx 1 wins.

Steady-state matches because both update by `(grant + 1) mod N` after a
grant. Severity: low (single-cycle skew) but it's a real SV↔sim divergence
the PR leaves unfixed.

**Proposal.** Initialise `_last_grant = N - 1` at reset *or* change the sim
scan to `(_last_grant + _i) % N` and only bump the pointer to
`(grant + 1) % N` on grant.

**Test that would have caught this.** A 3-master persistent-contention
fixture asserting strict grant sequence `0,1,2,0,1,2,…` for the first ~30
cycles, run under both default sim and Verilator with byte-identical
expectation.

## 3. `--thread-sim` silently downgrades mutex policy to priority

**Setting.** PR #443 relaxed the policy gate at
`src/sim_codegen/thread_sim.rs:147-153` to accept `RoundRobin` and `Lru`, but
the scheduler at lines 808-814 still does "free or already mine," which
resolves to slot-iteration-order (lowest thread index) — i.e. fixed
priority.

**Impact.** Under `arch sim --thread-sim ...`, `mutex<round_robin>` and
`mutex<lru>` are observationally identical to `mutex<priority>`. Anyone
using `--thread-sim both` for cross-check equivalence
(per `doc/COMPILER_STATUS.md`) on a design relying on RR fairness will get
false-PASS results.

**Proposal.** Emit a `--thread-sim` warning at codegen time when a
non-priority policy is in use:

```
warning: --thread-sim ignores mutex policy <P> on resource <name>;
         scheduler executes lowest-thread-index-wins (== priority)
```

or, longer-term, teach the `--thread-sim` scheduler to honour the policy.

**Doc gap.** `doc/thread_spec_section.md:201-260` and the AI reference card
both claim `mutex<round_robin>` distributes fairly with no mention of the
`--thread-sim` behaviour. `doc/thread_lowering_algorithm.md:66-75, 484-492`
describes only fixed-priority arbiters — already stale since v0.46.0 added
the policy menu; PR #443 doesn't refresh it.

## 4. nic400 AXI WRAP preconditions unasserted

**Setting.** PR #440 added FIXED/INCR/WRAP decode to `Nic400ApbBridge`; PR
#441 added FIXED-rejection assertions to `Nic400WidthAdapter`.

**Findings.**

- WRAP math (`tests/nic400/Nic400ApbBridge.arch:118`) matches AXI §A3.4.1 and
  verifies by hand for the two TB scenarios. But AXI4 requires WRAP `axlen
  ∈ {1,3,7,15}` and base aligned to `(1<<size)`; the new asserts at lines
  267-273 only catch reserved `0b11`. An illegal WRAP with `axlen=2` would
  silently compute `burst_bytes=12, mask=0xB` — UNPREDICTABLE in the spec,
  wrong in the bridge. Same gap in WidthAdapter `:297-298`.
- WidthAdapter upsize WRAP direction (`M_DATA_W < S_DATA_W`) is unverified
  by the new TB — only downsize is exercised.

**Proposal — TB additions, no compiler change.**

1. Illegal WRAP `axlen=2` (3-beat) → assert SVA fires or capture defined
   behaviour.
2. Unaligned WRAP base (`ar_addr=0x8003, size=2`) → capture observable.
3. WidthAdapter upsize WRAP mirror test (`M_DATA_W=32, S_DATA_W=64`,
   `m_ar_len=3` → `s_ar_len=1`).
4. 4 KB boundary crossing under INCR (`addr=0xFF8, len=3, size=2`) on APB
   bridge — either assert or document.

## 6. harc-com: RAL active-vs-passive callback asymmetry undocumented

**Setting.** PR #310 added `regs.record_write(...)` (passive) and `on
regs.REG` callbacks. The active frontdoor (`regs.A = v`) does **not** fire
callbacks; only `record_write` does.

**Impact.** A user writing a checker that mixes both paths will silently
miss events on the active side. The asymmetry is not in
`docs/ral-support.md`.

**Proposal (doc + test, no codegen change).**

1. Document the asymmetry in `docs/ral-support.md` §3.2 explicitly, plus:
   - re-registering `on regs.REG` silently overwrites;
   - unmapped `record_write`/`record_read` are no-ops returning 0;
   - callback recursion has no guard.
2. Add fixture cases: callback that writes its own register (recursion);
   active write that should-not-fire-callback; unmapped-address
   record_read.
3. Tech-debt watch: mirror-update logic is now duplicated across three
   sites in `src/codegen/cpp_tb.rs` (active path, field-level frontdoor,
   passive record). Before the next RAL slice (field callbacks, addrmap
   record), extract a single
   `emit_mirror_update_and_dispatch(...)` helper.

## 7. harc-com: testbench lifecycle parser invariant is implicit

**Setting.** PR #306 added `setup` / `check` / `teardown` blocks. Each
keyword produces its own `ComponentItem::Lifecycle(ScopeDecl)` AST node
with exactly one populated field (`src/parser.rs:1633-1689`). Duplicate
detection at `:1670` walks earlier `Lifecycle` nodes per-field.

The "one populated field per node" invariant is load-bearing for the
duplicate-check, the round-trip ordering, and the splice in
`merge_lifecycle_blocks`. If anyone collapses these into a single node
later, the duplicate check silently breaks.

The synthetic `name: "sim"` string stuffed into the `ScopeDecl` at three
sites is also a load-bearing magic value.

**Proposal.** Replace the field-overload with a tighter shape, e.g.

```rust
enum LifecyclePhase { Setup, Check, Teardown }
ComponentItem::Lifecycle(LifecyclePhase, Block)
```

Smaller, type-checked, and the duplicate check becomes
`existing.iter().any(|i| matches!(i, Lifecycle(p, _) if p == phase))`.

**Adjacent fixture gap.** No test for source-order independence (`setup`
declared after `check`); no test for field-rewrite of `_tb.X` inside
lifecycle blocks under the bare-statement impl branch.

## 8. harc-com: transactor topo sort under-approximates

**Setting.** PR #309 added `topo_sort_component_indices` for emission
order. The dependency graph only walks `ComponentItem::Field` references
(`src/codegen/cpp_tb.rs:2201-2205`). PR #308 makes transactor-local
hookable calls from `on`-handler bodies first-class.

**Latent risk.** A call-dependency that is not mirrored by a field-type
dependency (e.g. C calls `A_method` from a free hook registered at test
scope, with no `field a: A` in C) is invisible to the topo sort and will
re-introduce forward-ref bugs.

**Proposal.**

1. Add a fixture: cross-transactor `on`-handler call where the callee is
   *not* a field of the caller. Today this works only by source-order
   accident.
2. Either extend `topo_sort_component_indices` to walk hookable-call
   targets, or document that all cross-transactor calls must go through a
   field.

Also: dead loop `for _ in d {}` at `src/codegen/cpp_tb.rs:2217-2225` with
a misdescribing comment ("Each dep j → i adds one incoming edge to i" —
`deps[i]` is i's *out*edges). Remove or clarify.

## A. Synthesized port-regs trigger `check_port_reg_timing` false positives

**Discovered 2026-05-28** while reviewing arch-com#454's build log:
the `arch check` pass on `Nic400WidthAdapter.arch` emits four
"registered output assigned inside a state-dependent branch" warnings
naming `rd_axlen_r`, `wr_axlen_r`, `r_pack_r`, `r_resp_r`. All four
are declared in source as plain internal `reg` (line 79–91), not as
`port reg` or `port: out pipe_reg<T, N>`. The user can't act on the
warning without changing their source in a way that doesn't fit the
design.

**Root cause.** Thread lowering at `src/elaborate.rs:2368-2382` walks
every register assigned in a seq block of a thread and synthesizes a
registered output port for it on the internal `_<Module>_threads`
submodule. The synthesized port reg carries `legacy_port_reg: false`
(to suppress the *deprecation* warning per the comment "don't
deprecate internal artifacts"), but the newer
`check_port_reg_timing` warning at `src/typecheck.rs:1812` has no
analogous gate — it fires on every registered output port regardless
of provenance.

**Proposal.** Add a `synthesized: bool` field to `PortRegInfo` in
`src/ast.rs:351`. Set it `true` at the thread-lowering site
(`elaborate.rs:2368`), `false` at parser sites
(`parser.rs:1400`). Gate `check_port_reg_timing` on
`!ri.synthesized`. ~10 lines. No spec change — purely an internal
flag, no observable behaviour change beyond suppressing the false
positive.

## C. Custom-policy hook liveness obligation (doc-fixed)

Discovered while refreshing `thread_lowering_algorithm.md` for the
v0.46.0 policy menu (arch-com#455): the synthesized arbiter for
`mutex<custom>` trusts whatever one-hot grant mask the user-provided
`hook grant_select(...)` returns. Mutual exclusion still holds (the
arbiter enforces one-hot regardless), but liveness is **only as good
as the hook**:

- Hook returns all-zeros while requests are asserted ⇒ arbiter-side
  deadlock.
- Hook consistently picks the same thread ⇒ starvation for others.

PR #455 documents this in `thread_lowering_algorithm.md` §Liveness
per policy. The compiler doesn't statically check hook liveness —
that obligation is the user's, and the doc now states this
explicitly so users of `mutex<MyQosPolicy>` know the contract.

No code change needed.

## D. Multi-thread safety of `[&]` captures in cpp_tb (deferred)

`src/codegen/cpp_tb.rs` emits ~62 `[&]`-capturing C++ lambdas. Most
are correct under the default single-threaded sim (`EmitOpts::mt =
false`), but the `mt: true` topology (per-actor `std::thread` with
dual-barrier sync, already shipping per `cpp_tb.rs:1656-1706`) makes
the RAL callback closures and hookable-method registration lambdas
potentially race-prone if a worker thread ever calls back into them.

Tracked in
[harc-com#316](https://github.com/arch-hdl-lang/harc-com/issues/316),
deferred until `mt: true` shows >2× wall-clock speedup on a
representative testbench. That issue lays out three scope options
(RAL-only audit / full `[&]` audit / runtime-side `Synchronized<T>`
wrapper) plus the prep work needed alongside (ThreadSanitizer in CI,
a reproducible perf benchmark).
