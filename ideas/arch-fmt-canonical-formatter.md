# `arch fmt` — Canonical Source Formatter for .arch Files

**Status:** Proposal — not yet implemented  
**Date:** 2026-06-21  
**Repositories:** arch-hdl-lang/arch-com (primary); arch-hdl-lang/harc-com has `harc fmt` as the reference implementation

---

## Problem

ARCH is explicitly designed to be generated by LLMs from natural-language hardware descriptions. However, LLM output has high formatting variance:

- Indentation width fluctuates (2 spaces, 4 spaces, tabs)
- Alignment of port declarations and `end` keywords is inconsistent
- Blank-line conventions differ across sessions and models
- Operator spacing varies (`a+b` vs `a + b`)

This creates friction at every stage:

1. **Code reviews** include formatting noise that obscures logic changes
2. **Learning-store entries** (via `arch advise`) are indexed by diff content — formatting noise pollutes retrieval quality
3. **LLM self-correction loops** need to distinguish style edits from semantic fixes; without a formatter, the LLM can't distinguish the two
4. **CI gates** can't enforce style without a canonical formatter
5. **Multi-file designs** assembled from several LLM sessions often have inconsistent style within a single build

HARC already has `harc fmt` (a round-trip-targeting pretty-printer in `src/pretty.rs`). ARCH has no equivalent. This is a toolchain asymmetry with a clear fix.

---

## Proposal

Add an `arch fmt` subcommand that formats `.arch` source files to a single canonical style — no configuration options, similar to `gofmt` and `rustfmt --edition 2021`.

### CLI surface

```sh
arch fmt File.arch               # print formatted output to stdout
arch fmt File.arch -i            # reformat in-place
arch fmt File.arch --check       # exit 1 if file would change (CI gate)
arch fmt a.arch b.arch c.arch    # multiple files
arch fmt a.arch -i --check       # compatible: check AND reformat
```

### Canonical style rules (opinionated, single style)

| Element | Rule |
|---|---|
| Indent unit | 2 spaces |
| Block body | indented 2 spaces relative to keyword line |
| Trailing whitespace | stripped |
| Blank lines | one blank line between top-level constructs; no blank lines inside port/param blocks |
| Port declarations | `port name: direction Type;` — single space after colon |
| Operator spacing | `a + b`, `a <= b +% 1`, `a & mask`, etc. — spaces around binary ops |
| `end keyword Name` | on its own line, not indented relative to opening `keyword Name` |
| Comment placement | `//` inline comments preserved; `///` and `//!` doc comments preserved verbatim |
| `reg default:` block | directly before first `reg` that uses the default |
| `generate for` / `generate if` | same indentation rules as other blocks |
| Long lines | no forced wrapping in v1 — width is not capped; deferred |

The style rules are intentionally minimal and dictated by the existing idiomatic ARCH examples in `doc/examples/` and `tests/`. The goal is *one canonical form*, not user preference.

### Round-trip guarantee

`arch fmt` must satisfy: `parse(fmt(src))` ≅ `parse(src)` (same AST, modulo comment placement in degenerate edge cases). Violations are bugs. This property:

- Doubles as a parser regression test
- Makes `--check` mode reliable for CI (if it would change, the file is not canonical)
- Prevents the formatter from silently altering semantics

## Why This Matters

### LLM-generation loop closure

The typical ARCH workflow today is:

```
natural-language description
  → LLM generates .arch
  → arch check (type-check)
  → arch build (emit SV)
  → arch sim (test)
  → error → arch advise → LLM fix → loop
```

Without `arch fmt`, every LLM-generated file enters the learning store with ad-hoc formatting. The advise retrieval index is diff-based; formatting noise degrades BM25 ranking. With `arch fmt`:

```
LLM output → arch fmt → arch check → (store canonical form)
```

The formatter step normalizes before indexing, so the learning store accumulates clean diffs.

### Team workflows

Teams writing ARCH by hand benefit from the same normalization: a pre-commit hook running `arch fmt --check` stops formatting debates. A reviewer can run `arch fmt -i` to strip noise before reviewing logic changes.

### Symmetry with HARC

`harc fmt` is already in HARC's CLI table. Every task that does `harc fmt` on HARC sources but can't do the equivalent on ARCH sources has an asymmetric workflow. The two sister languages should have matching toolchains.

## Implementation Approach

The implementation is a pure pretty-printer over the existing AST — no new analysis, no new IR passes.

### New file: `src/pretty.rs`

Mirror the structure of HARC's `src/pretty.rs`. Walk the parsed AST (from `src/parser.rs` → `ast.rs`) and emit canonical source text into a `String` buffer. The grammar's block structure is extremely regular (`keyword Name ... end keyword Name` everywhere), making this tractable:

```rust
pub fn format_file(file: &ast::ArchFile) -> String {
    let mut p = Printer::new();
    for item in &file.items {
        p.print_top_level(item);
        p.blank_line();
    }
    p.finish()
}
```

The `Printer` struct carries the current indentation depth and a `String` output buffer. It does not need the symbol table or type information — this is a pure structural transform.

### New subcommand in `src/main.rs`

```rust
/// Format ARCH source file(s) to canonical style
Fmt {
    /// Input .arch file(s) to format
    #[arg(required = true)]
    files: Vec<PathBuf>,
    /// Reformat in-place (default: print to stdout)
    #[arg(short = 'i', long)]
    in_place: bool,
    /// Exit 1 if any file would change; do not modify files
    #[arg(long)]
    check: bool,
},
```

When `--check` is set without `-i`, report which files differ and exit 1 if any. When both are set, reformat and report.

### Round-trip test

Add to `tests/round_trip.rs` (which already exists for the pretty-printer path):

```rust
#[test]
fn fmt_round_trip() {
    // Parse → fmt → parse again; assert the two ASTs are equal.
    for fixture in glob::glob("tests/fixtures/**/*.arch").unwrap() {
        let src = std::fs::read_to_string(&fixture).unwrap();
        let ast1 = arch::parser::parse(&src).unwrap();
        let fmt1 = arch::pretty::format_file(&ast1);
        let ast2 = arch::parser::parse(&fmt1).unwrap();
        let fmt2 = arch::pretty::format_file(&ast2);
        assert_eq!(fmt1, fmt2, "formatter is not idempotent on {}", fixture.display());
    }
}
```

Idempotency (`fmt(fmt(x)) == fmt(x)`) is a sufficient condition for the round-trip guarantee and easy to test mechanically.

## Non-Goals

- **Style configuration** — no flags for indent size, line width, etc. Opinionated, single style, no debates.
- **Semantic changes** — the formatter never changes observable semantics (no dead-code removal, no reordering of declarations).
- **Auto-fix for type errors** — `arch fmt` only runs the parser, not the type checker. Ill-typed files are still formatted if they parse.
- **Automatic blank-line insertion** between declarations within a block — v1 preserves existing blank lines within blocks.
