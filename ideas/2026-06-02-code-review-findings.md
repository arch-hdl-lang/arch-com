## 1. Test gap (filed): split-shard build/link is unguarded by CI

**Status: PR filed ŌĆö harc-com#332.**

The #330 split codegen is covered only by string-level unit tests in
`tests/codegen.rs` (`split_tests_emit_dispatcher_and_one_unit_per_test`,
`grouped_split_reemits_per_test_suffixes`). They assert on generated file
names and `#include` lines but never compile or link the output.

The load-bearing property ŌĆö that the dispatcher TU plus N shard TUs link
into one working binary ŌĆö holds only because every re-emitted file-scope
definition (`HarcTestContext`, all `static`/template helpers, `harc_rng`)
has internal linkage, while the per-test `run_<Test>` functions (unique per
shard) and `main` (dispatcher only) are the sole external symbols. A future
refactor that promoted any scaffolding to external linkage, or desynced the
dispatcher's `extern int run_<Test>(...)` declarations from the emitted
definitions, would pass every existing test but break a real build with a
duplicate/undefined-symbol link error.

harc-com#332 adds `tests/split_build_e2e.rs`, which drives the real CLI
through Verilator for both the default-group and group-size-1 layouts and
asserts link + per-test dispatch. Gated on `verilator` being present.

## 2. Tech debt (FIXED ŌĆö harc-com#333): the header-factoring path served only the non-default config

`emit_split_tests_with_file_prefix` has two branches:

- `group_size > 1` (the **default**, group 4) ŌĆö each shard is
  `emit_with_opts(filtered_source)` + `strip_generated_dispatcher`, i.e. a
  fully self-contained TU. **No** `common_*.hpp` / `.inc` files are emitted.
- `group_size == 1` ŌĆö the elaborate common-prefix/suffix string-slicing
  machinery (`common_split_prefixes`, `common_line_prefix_len`,
  `common_line_suffix_len`, `split_single_test_cpp`, `emit_split_header`,
  `emit_split_include`) that factors byte-identical slices into generated
  headers and `.inc` snippets.

So the most intricate code in the PR is reachable **only** at
`--cpp-split-group-size 1`, which the PR's own benchmark shows is *slower*
than the default group 4. `docs/separate-compilation-plan.md` describes the
header/`.inc` factoring as the general mechanism, which is imprecise ŌĆö at
the default group size none of those files exist. The user-facing
`docs/harc-sim-cli.md` is correct (it promises only "a dispatcher `main.cpp`
+ one or more grouped shard `.cpp` files").

**Resolved in harc-com#333.** Both paths were collapsed into one
self-contained-shard loop: the factoring bought no build-time win (the
common header is compiled once per TU anyway) and gave identical
incremental granularity (each per-test `.cpp` is already byte-stable when
other tests change, so `write_if_changed` lets Make skip it ŌĆö verified end
to end). `group_size==1` now emits one `test_<name>.cpp` per test, no
`.hpp`/`.inc`. Net ŌłÆ241 lines in `cpp_tb.rs`; the unreachable
shard-filename `else` arm is gone; `separate-compilation-plan.md` rewritten
to match. Verified through Verilator for both group-1 and group-4 layouts.

## 3. Proposal: auto-discover `.archi` for **bus port types**, not just module `inst`

**Evidence.** `arch check tests/nic400/Nic400MasterPort.arch` (the exact
command in #488's test plan, listed as "OK") actually fails:

```
├Ś unknown bus type `BusAxi4`
   ŌĢŁŌöĆ[tests/nic400/Nic400MasterPort.arch:35:19]
35 Ōöé   port m:  target BusAxi4<ADDR_W=ADDR_WIDTH, ...>;
```

It only succeeds when the bus definition is passed explicitly:
`arch check tests/nic400/BusAxi4.arch tests/nic400/Nic400MasterPort.arch`
ŌåÆ `OK`. A `BusAxi4.archi` sits right next to the source, so the
separate-compilation system is clearly *meant* to expose bus interfaces.

**Root cause** (`src/main.rs`, the dep-discovery loop, ~L1422ŌĆōL1462): the
resolver scans for undefined references **only** in `ModuleBodyItem::Inst`
nodes inside `Module` and `Pipeline` bodies. A bus reference in a port
declaration ŌĆö `port m: target BusAxi4<...>` / `port m: initiator BusAxi4` ŌĆö
is never inspected, so `{BusName}.arch` / `{BusName}.archi` is never queued
from the same dir / `ARCH_LIB_PATH` / stdlib the way module insts are.

**Proposal.** Extend the discovery loop to also collect bus type names from
`initiator`/`target` port declarations across all constructs (not just
Module/Pipeline insts), and resolve `{BusName}.arch|.archi` through the same
search path. Purely additive: it makes single-file `arch check` of a
bus-consuming module succeed via auto-discovery instead of an "unknown bus
type" error; designs that already pass the bus file explicitly are
unaffected.

**Why it matters beyond ergonomics.** This gap is part of why #485's
regression slipped: a reviewer running the PR's own
`arch check <single-file>` command sees a confusing *unrelated* error
("unknown bus type") rather than the actual retired-syntax error, which
discourages running the single-file check at all. Making single-file check
self-sufficient strengthens the cheapest CI guard.

**Scope check (per `CLAUDE.md`).** This changes the *compilation model*
(what inputs `arch check` needs), so it is filed as a proposal rather than
an autonomous fix. It does **not** change any surface syntax, type rules,
or operator/lowering semantics ŌĆö `.archi` auto-discovery for `inst` is
already a documented feature; this extends the same mechanism to bus
ports. If accepted, the one-line behavior note belongs in the
separate-compilation section of the spec / reference card.
