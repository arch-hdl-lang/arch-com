# `arch fmt` — Canonical Source Formatter

**Date:** 2026-06-09
**Status:** Proposal / discussion

---

## Problem

ARCH is explicitly designed to be generated by LLMs from natural-language hardware
descriptions. In practice, every LLM agent (and every human developer) produces
slightly different style: inconsistent indentation depth, varying spacing around
`<-` / `->` / `<=` / `:`, mixed blank-line conventions between constructs, and
random capitalisation of hex literals. The repo already has multiple concurrent
agents (Claude instances, the `codex/*` pipeline) — CLAUDE.md calls out the
duplicate-work hazard this creates. Style divergence compounds the coordination
problem: a diff that is purely a style reformat obscures a substantive change,
and `git blame` becomes noisy.

ARCH has syntax highlighting (VSCode TextMate grammar, Vim), an MCP server with
`write_and_check`, and `arch advise` for learning retrieval — but no formatter.
SystemVerilog has Verible; Verilog has `verilog-format`. ARCH has nothing.

---

## Proposed Solution

Add `arch fmt` as a new top-level command:

```
arch fmt [--check] [--in-place] [--indent N] file.arch...
```

| Flag | Meaning |
|------|---------|
| `--check` | Exit non-zero if any file would change; print a diff but do not rewrite. CI-safe. |
| `--in-place` | Rewrite files in place (default: print to stdout). |
| `--indent N` | Indentation width in spaces (default: 2). |

Exit codes: `0` = all files already canonical (or `--in-place` succeeded), `1` =
files would change (`--check` mode), `2` = parse error.

---

## Why ARCH's Grammar Makes This Tractable

ARCH uses a keyword/end-keyword grammar with no braces and no significant
whitespace. The parser already produces a full AST. A formatter just needs to
walk that AST and emit tokens with canonical spacing — there is no ambiguity about
block structure to recover. Compare this to formatting C++ or SystemVerilog, where
preprocessor macros and macro-expanded tokens make a round-trip formatter
extremely fragile.

The existing `src/parser.rs` produces `SourceFile` → `Vec<TopLevelItem>`. Each
`TopLevelItem` is already fully parsed with type, params, ports, body items, and
the closing name. A pretty-printer that walks these nodes and applies canonical
rules is a new pass (~500–800 lines) with no changes to the existing pipeline.

---

## Canonical Rules (proposed first cut)

### Block structure
- Top-level constructs separated by exactly one blank line.
- No trailing whitespace on any line.
- File ends with exactly one newline.

### Indentation
- Each nested level adds `N` spaces (default 2).
- `param`, `port`, body items, and the `end keyword Name` footer all at the same
  indent level relative to the opening keyword.

### Ports and params (one per line)
```arch
module Foo
  param WIDTH: const = 8;
  port clk: in Clock<SysDomain>;
  port rst: in Reset<Sync>;
  port data_in:  in  UInt<WIDTH>;
  port data_out: out UInt<WIDTH>;
```
Type columns are NOT aligned (alignment is fragile when params change); a single
space after the colon is canonical.

### Operators
- One space on each side of binary operators: `a + b`, `a <= b`, `a == b`.
- No space between a unary prefix operator and its operand: `not valid`, `~x`.
- One space after commas; no space before.
- `<-` and `->` in inst connections: one space on each side.

### Numeric literals
- Decimal literals: no change.
- Hex literals: canonical lowercase prefix and digits: `0xff`, `8'hff`.
- Binary literals: `1'b0`, `1'b1`.
- Sized unsized literals: no space between size and base: `8'd200`.

### Comments
- `//` comments: preserve as-is (formatter does not reflow prose).
- `///` and `//!` doc comments: preserve as-is.
- `/* */` block comments: preserve (but not currently generated by any pass).

### `comb` and `seq` blocks
```arch
  comb
    out_valid = in_valid and not stall;
  end comb

  seq on clk rising
    data_r <= data_in;
  end seq
```
Body statements indented one level inside the block.

### `if / elsif / else` alignment
```arch
    if cond_a
      x <= 1;
    elsif cond_b
      x <= 2;
    else
      x <= 3;
    end if
```
The `elsif` and `else` keywords sit at the same indentation as the matching `if`.

### `match`
```arch
    match state_r
      Idle: x <= 0;
      Run:  x <= 1;
      _:    x <= 2;
    end match
```
Arm values are NOT column-aligned (fragile); one space after the colon.

### `inst` blocks
```arch
  inst sub: SubModule
    clk <- clk;
    rst <- rst;
    a_in  <- src_a;
    a_out -> dst_a;
  end inst sub
```

## Implementation Sketch

### New files
- `src/fmt.rs` — `pub fn format_source(src: &str, opts: FmtOpts) -> Result<String, CompileError>`
  - Calls the existing lexer + parser to get an AST.
  - Calls a new `Printer` struct that walks the AST and writes to an internal
    `String` buffer with indentation tracking.
  - Returns the formatted string.
- `src/main.rs` — new `Subcommand::Fmt` arm that reads file(s), calls `format_source`,
  and either prints to stdout, rewrites in place (`--in-place`), or diffs for CI
  (`--check`).

### Key design constraint: comment preservation

The parser currently drops comments. For a formatter to preserve comments, the
lexer must emit `Comment` tokens (not discard them) and the printer must
interleave them with AST nodes. Two options:

**Option A (simpler, lower fidelity):** Format from AST, strip all comments from
output. Only useful for LLM-generated code that has no comments. Reject this —
it would destroy human-authored doc comments.

**Option B (preferred):** Extend the lexer to keep comment tokens in a
side-channel (a `Vec<(ByteRange, CommentText)>` parallel to the token stream).
The printer reconstructs the output by interleaving comment blocks with AST nodes
using the original source byte offsets as anchors. This is the standard approach
used by `rustfmt`, `prettier`, and Verible.

The comment side-channel is a localized change to `src/lexer.rs` (~20 lines)
and a new helper in `src/fmt.rs` that reattaches comments to their preceding or
following AST node.

### Phased rollout
| Phase | Scope | Comment handling |
|-------|-------|-----------------|
| v1 | Format LLM-generated files (no comments) | Drop + warn if comments present |
| v2 | Full fidelity | Preserve via side-channel |
| v3 | `arch fmt --sort-ports` (alphabetize by direction then name) | Optional ordering pass |

## CI Integration

A single-line addition to the project's CI workflow:

```yaml
- name: ARCH format check
  run: arch fmt --check $(find . -name '*.arch')
```

This is the same pattern used by `gofmt -l`, `rustfmt --check`, and
`swift-format --lint`. It catches style drift from AI-generated commits before
they land.

## Why This Matters Now

1. **LLM generation quality**: Every time an agent generates ARCH from a spec, the
   output style differs slightly. A formatter means the commit diff shows only
   semantic changes.

2. **Multi-agent coordination**: CLAUDE.md §"Claim work before you start it" calls
   out that multiple agents work the same repo concurrently. Style normalization
   reduces merge conflicts caused by whitespace-only divergences.

3. **Review ergonomics**: Reviewers see consistent style in every PR; no mental
   overhead distinguishing "style change" from "logic change".

4. **Implementation fit**: ARCH's keyword/end-keyword grammar (no braces, no
   significant whitespace) makes a round-trip formatter far simpler than for SV.
   The existing parser already does the hard structural work.

5. **Gap in tooling**: VSCode extension, Vim syntax, MCP server, and `arch advise`
   are shipped. A formatter is the standard next step in any language toolchain and
   is the only common developer-experience tool ARCH is missing.

## Open Questions

1. **Default indent width**: 2 spaces (Go/ARCH community feel) or 4 spaces (Rust)?
   Suggest 2 since ARCH nests deeply (module → comb → if → elsif).

2. **Trailing comma normalisation**: Should the formatter add/remove semicolons on
   the last `param` or `port`? Current grammar requires semicolons; no change
   needed, but worth confirming.

3. **Long-line wrapping**: Should the formatter wrap lines exceeding 100 chars?
   V1 suggest no wrapping (LLMs produce short enough lines; wrapping is fragile).
   Defer to v3.

4. **Port alignment**: Some codebases align port types in columns. V1: no
   alignment (fragile on param changes). Document this decision explicitly so
   contributors don't fight it.

5. **`--diff` flag**: Print a coloured unified diff rather than the rewritten
   file — useful in CI to show exactly what changed. Can be added in v2.
