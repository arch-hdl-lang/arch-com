# Structured machine-readable diagnostic output (`arch check --format json`)

**Date:** 2026-06-11
**Status:** Proposal
**Area:** Compiler / Developer Experience / LLM workflow

---

## Problem

ARCH is explicitly designed to be generated by LLMs from natural-language hardware descriptions
(see spec introduction and `CLAUDE.md`). The core loop is:

```
LLM generates .arch source
  → arch check / arch build
  → parse error text
  → LLM applies fix
  → repeat
```

The "parse error text" step is the weak link. Errors currently use `miette` for rich terminal
output — beautiful for humans, fragile for machines. An LLM agent (or the MCP `write_and_check`
tool, or a CI script) must regex-parse stderr to extract the error kind, location, and hint. Any
change to the message text silently breaks these parsers.

The same fragility affects:
- `arch advise`: the learning store's error→fix classifier regex-matches on the raw error string
  to categorize events (`events.jsonl`); a structured error code would be more stable and precise.
- The VSCode extension: inline squiggles require line/column data extracted from the terminal
  output.
- CI pipelines: filtering "only width-mismatch errors" requires grepping message text.

## Proposed change

Add `--format json` (alias `--format jsonl`) to `arch check`, `arch build`, and `arch sim`.
When set, all diagnostics are written to stdout as a single JSON object (or one JSON object per
line for `jsonl`); human-readable `miette` output is suppressed.

### Output schema

```json
{
  "compiler_version": "0.70.4",
  "files": ["Foo.arch", "Bar.arch"],
  "diagnostics": [
    {
      "severity": "error",
      "code": "E_WIDTH_MISMATCH",
      "message": "width mismatch: target is 8 bits, value is 9 bits",
      "file": "Foo.arch",
      "span": { "line": 42, "col": 9, "end_line": 42, "end_col": 24 },
      "hint": "use .trunc<8>() to truncate the result"
    },
    {
      "severity": "warning",
      "code": "W_TODO_SITE",
      "message": "todo! site will abort at runtime if reached",
      "file": "Foo.arch",
      "span": { "line": 17, "col": 15, "end_line": 17, "end_col": 20 },
      "hint": null
    }
  ],
  "error_count": 1,
  "warning_count": 1,
  "exit_code": 1
}
```

Span line/col are 1-based to match editor conventions. `exit_code` mirrors the process exit
code, letting consumers check the JSON without also checking the OS exit status.

### Error codes

Each `CompileError` variant in `src/diagnostics.rs` gets a stable short code:

| Code | Variant |
|------|---------|
| `E_UNEXPECTED_TOKEN` | `UnexpectedToken` |
| `E_UNEXPECTED_EOF` | `UnexpectedEof` |
| `E_MISMATCHED_CLOSING` | `MismatchedClosingName` |
| `E_UNDEFINED_NAME` | `UndefinedName` |
| `E_UNDEFINED_MODULE` | `UndefinedModule` |
| `E_DUPLICATE_DEF` | `DuplicateDefinition` |
| `E_TYPE_MISMATCH` | `TypeMismatch` |
| `E_WIDTH_MISMATCH` | `WidthMismatch` |
| `E_MULTIPLE_DRIVERS` | `MultipleDrivers` |
| `E_UNDRIVE_OUTPUT` | `UndriveOutput` |
| `E_NAMING_VIOLATION` | `NamingViolation` |
| `E_LEXER` | `LexerError` |
| `E_GENERAL` | `General` (catch-all; add specific codes as the set grows) |

Warnings parallel this with `W_` prefix (`W_TODO_SITE`, `W_DEAD_SKID`, `W_CDC_MULTIBIT`, etc.).
`miette`'s `#[diagnostic(code = "arch::E_WIDTH_MISMATCH")]` already supports this — it's unused
today.

## Why it matters

### 1. LLM agent reliability

The MCP server's `write_and_check` returns raw stderr. An LLM parsing
```
error: width mismatch: target is 8 bits, value is 9 bits
  → src/Foo.arch:42:9
```
must extract line 42, col 9 by convention. With JSON output the MCP tool deserializes directly:

```rust
let result: DiagOutput = serde_json::from_str(&stdout)?;
for d in &result.diagnostics {
    if d.severity == "error" {
        // d.code, d.span.line, d.hint — all typed, no regex
    }
}
```

This matters most when errors are nested or span multiple files, where the current text format
becomes hard to delimit.

### 2. `arch advise` classification quality

The learning store records error→fix pairs keyed on the *text* of the error. `classify_error()`
in `src/learn.rs:975` is a cascade of `str.contains(...)` checks against the message string — a
fragile regex that covers only ~10 known patterns and falls through to `"other"` for anything
else. For example, a CDC error in a novel phrasing won't hit the `"clock domain"` branch.

With stable enum-level codes (`E_WIDTH_MISMATCH`, etc.) the function becomes a one-line map lookup
instead of pattern matching on text. All width-mismatch variants bucket under the same code
regardless of the message wording, and `arch advise` retrieves higher-quality suggestions across
the full error vocabulary.

### 3. IDE extension

The VSCode extension (`editors/vscode/`) currently does syntax highlighting only. Enabling inline
error markers (squiggly underlines) requires running `arch check` in the background and mapping
errors to editor positions. JSON output with 1-based line/col data makes this a 50-line
extension change.

### 4. Zero breaking change

Human-readable output (the default) is unchanged. `--format json` is additive; existing scripts
that parse stderr are unaffected unless they opt in.

## Implementation sketch

All changes are in `src/`:

### Step 1 — Add stable codes to `CompileError` (~30 min)

```rust
// src/diagnostics.rs
#[derive(Error, Debug, Diagnostic)]
pub enum CompileError {
    #[error("width mismatch: target is {target_width} bits, value is {value_width} bits")]
    #[diagnostic(code = "arch::E_WIDTH_MISMATCH")]
    WidthMismatch { ... }
    // ... add #[diagnostic(code = "arch::E_...")] to every variant
}
```

### Step 2 — Add `OutputFormat` enum and CLI flag (~2 hours)

```rust
// src/main.rs (or a new src/output_format.rs)
#[derive(Clone, Debug, PartialEq)]
pub enum OutputFormat { Human, Json, JsonL }
```

Add `--format <human|json|jsonl>` to the top-level clap CLI, default `human`.

### Step 3 — JSON serialization layer (~2-3 hours)

Add a `DiagOutput` struct that's `serde::Serialize`:

```rust
#[derive(Serialize)]
pub struct DiagOutput<'a> {
    compiler_version: &'static str,
    files: Vec<&'a str>,
    diagnostics: Vec<DiagEntry>,
    error_count: usize,
    warning_count: usize,
    exit_code: i32,
}

#[derive(Serialize)]
pub struct DiagEntry {
    severity: &'static str,   // "error" | "warning"
    code: String,
    message: String,
    file: Option<String>,
    span: Option<DiagSpan>,
    hint: Option<String>,
}

#[derive(Serialize)]
pub struct DiagSpan {
    line: u32, col: u32,
    end_line: u32, end_col: u32,
}
```

Convert `CompileError` / `CompileWarning` instances to `DiagEntry` by extracting the code from
`miette::Diagnostic::code()` and converting the `SourceSpan` to line/col via the existing source
map.

### Step 4 — Wire into each subcommand exit path (~1 hour)

Currently `arch check` / `arch build` / `arch sim` print `miette` errors and exit. Wrap the
existing error-collection path: if `--format json`, collect into `DiagOutput`, serialize to
stdout, and skip `miette` rendering.

### Step 5 — MCP server update (~30 min)

`write_and_check` in the MCP server passes `--format json` and deserializes the output instead
of regex-parsing stderr. Returns structured `{ errors: [...], warnings: [...] }` to callers.

### Step 6 — Tests (~1-2 hours)

Add integration tests that run `arch check --format json` on known-bad `.arch` files and assert
the JSON contains the expected `code`, `span.line`, and `hint` values.

### Total estimated effort: ~1.5–2 days

No parser, elaboration, or codegen changes required. The `diagnostics.rs` enum and `main.rs`
entry points are the only affected source areas.

## Dependencies

- `serde` + `serde_json` — already in the Rust ecosystem; check `Cargo.toml` before adding
- `miette`'s `code()` method on `dyn Diagnostic` — already available; currently unused
- Line/col conversion from `SourceSpan` — needs a helper that walks the source text; a
  one-pass scan over `\n` occurrences is sufficient

## Non-goals

- **LSP / language server protocol**: a full LSP would reuse the JSON output as one component,
  but is a much larger effort. This proposal only covers the diagnostic serialization layer that
  an LSP would eventually consume.
- **Interactive REPL**: not related.
- **Error recovery / multiple errors per parse**: the compiler already emits all errors it finds;
  this proposal only changes their output format.

## Relationship to existing work

- `arch advise` / learning system (`plan_arch_learning_system.md`): the structured error codes
  make classification more robust; no changes needed to the learning store schema.
- MCP server `write_and_check`: a direct beneficiary; the MCP update is a small follow-on.
- VSCode extension (`editors/vscode/`): enables inline error markers as a follow-on PR.
- `--inputs-start-uninit` / `--check-uninit` warnings: these produce `CompileWarning` structs
  which follow the same path; they'd be included in `diagnostics` with `severity: "warning"`.

## Open questions

1. **`--format jsonl` vs `--format json`**: `jsonl` (one object per line) is easier to stream
   from long-running `arch sim` invocations; `json` (single object) is simpler for `arch check`.
   Recommend starting with `json` only and adding `jsonl` when `arch sim` needs streaming output.

2. **Span source for General errors**: many `General` errors don't have precise spans. Emit
   `"span": null` rather than a fake span. The code still conveys the error category.

3. **Backwards compatibility promise**: once error codes are assigned, changing them is a
   breaking change for any tooling that matches on them. Recommend a brief review pass before
   shipping to assign codes that reflect the actual semantic category, not just the Rust enum
   variant name.
