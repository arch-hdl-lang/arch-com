# Zero-TB Fuzz Mode: `arch sim --randomize-inputs`

**Status**: Proposal  
**Date**: 2026-06-28  
**Affects**: `arch sim`, sim codegen (`src/sim_codegen/`), `src/main.rs`

---

## Problem

Every `arch sim` run today requires a hand-written testbench (`--tb tb.cpp` or
`--pybind --test foo.py`).  This is reasonable for integration tests, but it
creates a barrier for two high-value early-stage workflows:

1. **LLM-generated designs need fast smoke-testing.**  When an LLM produces a
   module from a natural-language spec, the first question is: "do the `assert`
   properties I can statically express about this design hold?"  Writing a
   testbench to drive all ports just to reach that answer takes minutes of
   boilerplate that is unrelated to the design's correctness.

2. **Regression fuzz after incremental edits.**  A developer who adds a register
   or changes a state-machine transition wants to quickly re-check that existing
   `assert`/`cover` properties still hold before committing.  A full testbench
   suite may not exist yet.

The `arch formal` BMC path partially covers this, but it is currently limited to
flat modules with scalar types — no `Vec`, no `struct`, no `enum` at instance
boundaries.  `arch sim` supports all of those, but only when driven by a TB.

---

## Proposal

Add `--randomize-inputs` to `arch sim`.  When set, `arch sim` auto-generates a
synthetic C++ testbench that:

1. **Applies reset** for `--reset-cycles N` (default 4) clock cycles with
   `rst` asserted.
2. **Each cycle**, drives every scalar `in` port (excluding `Clock` and `Reset`
   ports) with a pseudo-random value within the port's declared type range:
   - `UInt<W>`: uniform random in `[0, 2^W)`.
   - `SInt<W>`: uniform random in `[-2^(W-1), 2^(W-1))`.
   - `Bool`/`Bit`: random 0 or 1.
   - `Vec<T,N>` `in` ports: each element driven independently (excluded in v1 —
     see Scope).
   - Bus `in` signals: each flattened scalar driven independently.
3. **After each `eval()`**, aborts with a trace if any `assert` property fires
   (already captured as the C++ model's `_arch_assert_*` hooks).
4. **Stops** after `--cycles K` (default 1000) cycles, or earlier if all
   `cover` points are hit.
5. **On assertion failure**, dumps `--wave fuzz_fail.vcd` pointing at the
   failing cycle so it can be inspected in GTKWave/Surfer.

### CLI

```bash
# Smoke-test a module for 2000 random cycles:
arch sim --randomize-inputs Adder.arch

# Longer run with fixed seed (reproducible):
arch sim --randomize-inputs --seed 42 --cycles 10000 Alu.arch

# Combine with --debug for port change tracing:
arch sim --randomize-inputs --debug Fifo.arch

# Dump waveform even on success (for visual inspection):
arch sim --randomize-inputs --wave out.vcd Fsm.arch
```

No `--tb` argument is accepted when `--randomize-inputs` is set.  The two modes
are mutually exclusive.

## Why This Matters

### For LLM-generated designs

ARCH is explicitly designed for LLM-generated RTL.  Today's validation loop is:

```
LLM generates module → arch check → write TB → arch sim
```

With this feature:

```
LLM generates module → arch check → arch sim --randomize-inputs
```

The LLM can embed `assert` properties in the module body that encode its
correctness intent (invariants, protocol constraints, data relationships), and
the zero-TB path immediately tests them without any additional authoring.  This
makes `assert`-annotated ARCH designs self-testable out of the box.

### For the formal–simulation gap

`arch formal` covers flat scalar modules; `arch sim` covers the full language.
`--randomize-inputs` fills the space between the two:

| Mode | Scope | Completeness | Requires TB |
|------|-------|--------------|-------------|
| `arch formal` | Scalar, flat | Complete (bounded) | No |
| `arch sim --randomize-inputs` | Full language | Probabilistic | No |
| `arch sim --tb` | Full language | User-defined | Yes |

Random sim will not catch every bug, but it is vastly better than no testing,
and it is far faster to reach than the formal backend on a 50-module hierarchy.

### For regressions

After a refactor, `arch sim --randomize-inputs --seed 42` can be wired into CI
as a quick sanity sweep.  The same seed on the same design produces the same
stimulus sequence, so a regression is reproducible.

## Implementation Approach

### Codegen path (`src/sim_codegen/`)

Minimal change to existing sim codegen.  The synthetic TB is a short C++ file
generated at sim-build time, identical in shape to a hand-written TB:

```cpp
// auto-generated by arch sim --randomize-inputs
#include "VModule.h"
#include <cstdint>
#include <cstdlib>
#include <cstdio>

int main(int argc, char** argv) {
    uint64_t seed  = /* --seed value or time(NULL) */;
    int      reset = /* --reset-cycles, default 4 */;
    int      total = /* --cycles, default 1000 */;

    // xorshift64 PRNG (no stdlib rand() — reproducible across platforms)
    auto rng = [&]() -> uint64_t {
        seed ^= seed << 13; seed ^= seed >> 7; seed ^= seed << 17;
        return seed;
    };

    VModule dut;

    // Reset phase
    dut.clk = 0; dut.rst = 1;
    for (int i = 0; i < reset; i++) {
        dut.clk = 0; dut.eval();
        dut.clk = 1; dut.eval();
    }
    dut.rst = 0;

    // Fuzz phase
    for (int cycle = 0; cycle < total; cycle++) {
        // Drive each in port with a random value in range
        dut.port_a = rng() & ((1ULL << W_A) - 1);  // UInt<W_A>
        dut.port_b = rng() & ((1ULL << W_B) - 1);  // UInt<W_B>
        // ... (generated per-port) ...

        dut.clk = 0; dut.eval();
        dut.clk = 1; dut.eval();

        // arch_assert_* hooks fire inside eval() on failure — no extra check needed
    }

    printf("arch sim --randomize-inputs: %d cycles, no assertion failures\n", total);
    return 0;
}
```

The per-port random driver is generated by inspecting each port's type in the
AST, which the sim codegen already traverses.  Width masks are already computed
for `--check-uninit` and `--inputs-start-uninit` paths, so the range arithmetic
is reused.

### Assertion wiring

The C++ models emitted by `arch sim` today call `_arch_assert_fail(...)` and
`abort()` whenever a user-written `assert` property fires.  `--randomize-inputs`
does not need to add any new checking logic — the existing abort path handles it.
The only addition is the VCD dump hook: if the process was about to abort, first
flush the VCD buffer and report the failing cycle number.

### Flag plumbing (`src/main.rs`)

Add `--randomize-inputs` (bool), `--seed` (u64, default 0 → use elapsed nanos
at startup), `--reset-cycles` (u32, default 4), and `--cycles` (u32, default
1000) to the `sim` subcommand's `clap` arg list.  When `--randomize-inputs` is
set, generate the synthetic TB instead of requiring `--tb`.

## Scope and Exclusions (v1)

**In scope:**
- Scalar `in` ports: `UInt<N>`, `SInt<N>`, `Bool`, `Bit`
- Bus `in` signals (flattened scalars)
- Single-clock modules

**Deferred (v2 and beyond):**
- `Vec<T,N>` `in` ports — need element-wise driving; straightforward extension
- `struct` `in` ports — same element-wise pattern
- Multi-clock modules — need independent random drivers per domain
- Constrained random (user-supplied `keep` constraints, like HARC) — out of scope
  for ARCH, which has no constraint language on the HDL side
- Coverage-guided mutation (greybox fuzzing) — possible future direction

**Non-goals:**
- Replacing `arch formal` — random sim is probabilistic; BMC is complete within
  the bound.  Both should be used.
- Replacing hand-written TBs — protocol-correct stimulus (AXI handshaking,
  UART framing, etc.) still requires a TB.  `--randomize-inputs` is a
  supplement, not a replacement.

## Related Work and Rationale for ARCH

This pattern is well-established in software fuzzing (AFL, libFuzzer) and formal
hardware (random simulation in Verilator + assertions has been a standard CI
pattern for years).  The ARCH-specific motivation is:

- ARCH's strict type system means random values are always in-range by
  construction (no Verilog-style X/Z or width-slipping).
- ARCH's `assert` keyword is the natural oracle — it lives in the design, not in
  a separate testbench.
- The sim codegen already emits type-width-aware C++ — the random driver is
  straightforward to generate from the same type information.

The result is a "press one button to test your module" experience that is
particularly natural for LLM-generated designs where the LLM can be prompted to
include `assert` invariants inline.

## Open Questions

1. **Waveform on success?**  By default, only dump VCD on failure.  Add
   `--wave` to force always-on tracing (already a flag, composes cleanly).

2. **Cover stopping condition?**  When all `cover` targets are hit before
   `--cycles` expires, should the run stop early and print a "full coverage hit"
   message?  Probably yes — makes the mode useful for completeness-checking too.

3. **`arch sim --randomize-inputs` in CI?**  The seed-based reproducibility
   makes this CI-friendly, but the default `--seed 0` (time-seeded) would be
   non-deterministic.  Recommendation: CI scripts should always pass `--seed`.

4. **SInt range**: Random `SInt<W>` values include the full signed range.  For
   designs where signed overflow is meaningful, this is correct.  For designs
   where the upper range is invalid by protocol, users should add `assert`
   properties to document the invariant — which is then immediately testable.
